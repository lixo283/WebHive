# WebHive Studio


## Опубликованный проект

Сайт доступен на Render:

**https://webhive.onrender.com**

Render используется как учебный хостинг для демонстрации проекта. На бесплатном тарифе сервис может "засыпать", поэтому первый запрос после простоя иногда открывается дольше обычного.

## Краткое описание

WebHive Studio имитирует работу небольшой веб-студии. Пользователь может посмотреть услуги, изучить портфолио, зарегистрироваться, авторизоваться и оставить заявку на выбранную услугу. После отправки заявки клиент видит ее в личном кабинете и отслеживает статус.

Администратор через админ-панель управляет услугами, портфолио и заявками. Для заявки можно поменять статус, указать финальную цену после обсуждения технического задания и оставить комментарий менеджера. Это закрывает важный бизнес-вопрос: базовая цена показывается в каталоге, а окончательная стоимость фиксируется после согласования деталей с клиентом.

## Стек

- Frontend: HTML5, CSS3, Vanilla JavaScript
- Backend: Node.js, Express.js
- Database: PostgreSQL
- SQL driver: `pg`
- Auth: JWT + bcrypt
- Deploy: Render
- Design: Figma
- Version control: Git + GitHub

- Просмотр истории статусов.

- Фильтрация заявок по статусу.
- Открытие деталей заявки прямо под выбранной строкой.

- Указание финальной цены.
- Комментарий менеджера к расчету или следующему шагу.

## Роли пользователей

| Роль | Возможности |
| --- | --- |
| Гость | Смотрит публичные страницы, каталог, портфолио и контакты |
| Пользователь | Регистрируется, входит в систему, оставляет заявки, смотрит личный кабинет |
| Администратор | Управляет услугами, портфолио и заявками |

## Как работает заявка

1. Клиент выбирает услугу в каталоге.
2. Переходит на детальную страницу услуги.
3. Авторизуется или регистрируется.
4. Заполняет контактные данные и комментарий.
5. Backend валидирует поля и создает заявку.
6. В заявке сохраняется выбранная услуга, контакты клиента и стартовый статус `new`.
7. Для заявки создается первая запись в истории статусов.
8. Администратор видит заявку в админ-панели.
9. После обсуждения ТЗ администратор указывает финальную цену и комментарий.
10. Клиент видит итоговый расчет в личном кабинете.

## Логика цены

В каталоге указана базовая цена услуги. Она нужна для ориентира клиента до начала общения.

Финальная цена хранится отдельно в поле `final_price` таблицы `applications`. Ее указывает администратор после обсуждения:

- объема работ;
- сроков;
- состава страниц;
- требований к дизайну;
- интеграций;
- правок и сопровождения.

Если финальная цена еще не указана, интерфейс показывает текст "После согласования ТЗ". Это честнее, чем притворяться, что у всех проектов одинаковая стоимость.

## Безопасность

В проекте реализованы базовые меры безопасности для учебного веб-сервиса:

- пароли хранятся как bcrypt-хеши;
- JWT-секрет проверяется при запуске и не может быть коротким дефолтным значением;
- защищенные API требуют авторизации;
- admin-операции требуют роли `admin`;
- SQL-запросы выполняются через параметризованные запросы;
- пользовательский ввод валидируется на backend;
- телефон заявки нормализуется и проверяется в формате `+7 (999) 999 99 99`;
- email проверяется регулярным выражением;
- добавлены HTTP-заголовки `Content-Security-Policy`, `Referrer-Policy`, `X-Content-Type-Options`;
- HTML, CSS и JS отдаются с `Cache-Control: no-cache`, чтобы Render не держал старую версию интерфейса.

## База данных

Проект использует PostgreSQL. Основные таблицы:

| Таблица | Назначение |
| --- | --- |
| `users` | Пользователи и роли |
| `services` | Услуги и тарифы веб-студии |
| `portfolio` | Кейсы портфолио |
| `applications` | Заявки клиентов |
| `application_status_history` | История изменения статусов |

Связи:

- один пользователь может иметь много заявок;
- одна услуга может быть выбрана во многих заявках;
- один проект портфолио может быть связан с услугой;
- одна заявка имеет много записей истории статусов.

## Быстрый запуск локально

В корне проекта:

```bash
./up.sh
```

Скрипт выполняет подготовку окружения:

1. Проверяет наличие `node`, `npm` и `psql`.
2. Создает `backend/.env`, если файла нет.
3. Генерирует безопасный `JWT_SECRET`, если он отсутствует.
4. Устанавливает backend-зависимости.
5. Проверяет подключение к PostgreSQL.
6. Создает базу `webstudio`, если она отсутствует.
7. Применяет `database/schema.sql`.
8. Применяет `database/seed.sql`.
9. Запускает backend.

После запуска сайт доступен локально:

```text
http://localhost:3000
```

## Ручной запуск

```bash
cd backend
npm install
cp .env.example .env
```

В `backend/.env` нужно указать настройки PostgreSQL и безопасный `JWT_SECRET` длиной от 32 символов.

Применение схемы и seed-данных:

```bash
psql -U postgres -d webstudio -f ../database/schema.sql
psql -U postgres -d webstudio -f ../database/seed.sql
```

Запуск backend:

```bash
npm run dev
```

| --- | --- | --- | --- |

## Ограничения учебной версии

- Оплата не подключена, потому что проект демонстрирует заявку и согласование стоимости, а не платежный шлюз.
- Почтовые уведомления не подключены.
- Файловая загрузка изображений в админке не реализована, используются пути к уже подготовленным изображениям.
- На бесплатном Render возможна задержка первого открытия после простоя.
