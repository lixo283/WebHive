## 1. Название проекта

**Веб-сервис «Веб-студия»**

---

## 2. Описание предметной области

Веб-студия занимается созданием и сопровождением цифровых продуктов для клиентов: сайтов, посадочных страниц, корпоративных порталов, редизайна интерфейсов и технической поддержки. Основной бизнес-процесс студии включает прием заявок, уточнение требований, оценку стоимости, выполнение работ и дальнейшее сопровождение.

### Какие услуги предоставляет веб-студия

- Разработка сайтов «под ключ»
- Дизайн интерфейсов и UX-аудит
- Верстка и фронтенд-реализация
- Интеграция форм и базовой бизнес-логики
- Техническая поддержка и обновление контента
- SEO-базовая оптимизация и повышение производительности

### Зачем нужен сайт веб-студии

Сайт выступает как единая цифровая площадка для презентации услуг, кейсов и условий сотрудничества, а также как канал для онлайн-взаимодействия с клиентами.

### Какие проблемы решает сайт

- Уменьшает долю ручной обработки заявок
- Ускоряет первичную коммуникацию с клиентами
- Систематизирует каталог услуг и тарифов
- Позволяет отслеживать статус заявок в личном кабинете
- Повышает прозрачность работы студии за счет портфолио и истории проектов

## 3. Цель проекта

Цель проекта — разработать веб-сервис «Веб-студия», который автоматизирует взаимодействие студии с клиентами и централизует ключевые процессы.

### Подцели

- Автоматизация приема и обработки заявок
- Создание структурированного каталога услуг и тарифов
- Реализация онлайн-подачи заявок без офлайн-каналов
- Управление статусами заявок на стороне администратора
- Публикация портфолио выполненных проектов

## 6. Сущности проекта

### 6.1 Контентная сущность: Услуга / тариф

Сущность хранит информацию о предлагаемых услугах студии и тарифных планах.

**Основные атрибуты:**

- Название услуги
- Краткое описание
- Полное описание
- Категория
- Базовая стоимость
- Срок выполнения
- Статус публикации

### 6.2 Дополнительная сущность: Проект портфолио

Сущность предназначена для публикации реализованных кейсов.

**Основные атрибуты:**

- Название проекта
- Клиент/сфера
- Описание задачи
- Описание решения
- Использованные технологии
- Изображение превью
- Ссылка на проект (при наличии)
- Дата завершения

### 6.3 Операционная сущность: Заявка на разработку

Сущность фиксирует обращение клиента и этапы обработки.

**Основные атрибуты:**

- Пользователь-инициатор
- Выбранная услуга
- Контактные данные
- Комментарий/техническое задание
- Желаемые сроки
- Бюджет (опционально)
- Текущий статус
- Дата создания/обновления

## 7. Структура базы данных

Ниже приведена логическая структура таблиц и связей (без SQL-реализации).

### 7.1 Таблица `users`

**Назначение:** хранение аккаунтов пользователей и администраторов.

**Поля:**

- `id` (PK)
- `name`
- `email` (уникальное)
- `password_hash`
- `role` (user/admin)
- `created_at`
- `updated_at`

### 7.2 Таблица `services`

**Назначение:** хранение каталога услуг и тарифов.

**Поля:**

- `id` (PK)
- `title`
- `slug`
- `short_description`
- `full_description`
- `category`
- `price_from`
- `duration_days`
- `is_active`
- `created_at`
- `updated_at`

### 7.3 Таблица `portfolio`

**Назначение:** хранение кейсов/проектов студии.

**Поля:**

- `id` (PK)
- `title`
- `client_name`
- `industry`
- `task_description`
- `solution_description`
- `tech_stack`
- `image_path`
- `project_url`
- `completed_at`
- `is_published`
- `created_at`
- `updated_at`

### 7.4 Таблица `applications`

**Назначение:** хранение заявок пользователей.

**Поля:**

- `id` (PK)
- `user_id` (FK -> `users.id`)
- `service_id` (FK -> `services.id`)
- `contact_name`
- `contact_email`
- `contact_phone`
- `comment`
- `deadline_preference`
- `budget`
- `status` (new/in_progress/approved/rejected/completed)
- `created_at`
- `updated_at`

### 7.5 Связи между таблицами

- `users` 1:M `applications` (один пользователь может создать много заявок)
- `services` 1:M `applications` (одна услуга может иметь много заявок)
- `portfolio` — независимая контентная таблица, управляемая администратором

## 8. Пользовательские сценарии

1. **Регистрация:** новый пользователь создает аккаунт по имени, email и паролю.
2. **Авторизация:** пользователь входит в систему и получает доступ к личному кабинету.
3. **Просмотр каталога:** гость или пользователь открывает список услуг и изучает предложения.
4. **Фильтрация услуг:** пользователь применяет фильтры по категории, цене и сроку.
5. **Отправка заявки:** авторизованный пользователь заполняет форму и отправляет заявку на услугу.
6. **Просмотр статуса:** пользователь в разделе «Мои заявки» отслеживает текущий статус обращения.
7. **Работа администратора:** администратор редактирует услуги и портфолио, обрабатывает входящие заявки, изменяет статусы.

## 12. Структура проекта

Ниже представлена планируемая структура каталогов и файлов проекта (на уровне документации):

- `docs/` — проектная документация

- `backend/routes/` — API-роуты
- `backend/controllers/` — бизнес-логика API

## 13. Дизайн-концепция

### Визуальное направление

Современный, чистый и профессиональный стиль с акцентом на доверие, кейсы и конверсию в заявку.

### Цветовая палитра

- Основной: глубокий синий/графит для акцентных элементов
- Дополнительный: светло-серые нейтральные фоны
- Акцентный: контрастный цвет для CTA-кнопок
- Текст: высококонтрастные оттенки для читаемости

### Стиль интерфейса

- Карточная подача услуг и кейсов
- Четкая типографическая иерархия
- Визуальные акценты на действиях «Оставить заявку»

### Адаптивность

- Поддержка desktop, tablet и mobile
- Адаптивная сетка и масштабирование контента
- Удобная навигация на мобильных устройствах

### UI-kit

- Базовый набор компонентов: кнопки, поля ввода, селекты, карточки, бейджи статусов, таблицы
- Единые состояния элементов: hover/focus/disabled/error/success

### Ключевые блоки

- Hero-блок на главной с ценностным предложением и CTA
- Секции «Услуги», «Портфолио», «Преимущества», «Этапы работы», «Контакты»

## Итог практики (ожидаемый результат)
