# Требования к проекту «Веб-студия»

## 0. Соответствие учебному ТЗ (обязательные условия)

1. Проект рассматривается как полноценный веб-сервис «Веб-студия», а не только как персональный сайт-портфолио.
2. Визуальная стилистика может быть выполнена как современное premium portfolio, но функционально проект обязан сохранять все сущности и роли из ТЗ.
3. Обязательные сущности:
- Контентная: услуги/тарифы (например, Landing Page, Интернет-магазин, Корпоративный сайт, UI/UX redesign).
- Дополнительная: проекты портфолио (изображение, описание, стек, категория).
- Операционная: заявки на разработку (отправка, статусы, история заявок).
4. Обязательные роли: гость, пользователь, администратор.
5. Обязательные технологические ограничения:
- Использовать: HTML5, CSS3, Vanilla JavaScript, Node.js, Express.js, PostgreSQL, Git, Figma.
- Не использовать: PHP, MySQL, PDO, React, Vue, Next.js, NestJS, Supabase, Laravel, Bootstrap, Tailwind, jQuery.

## 1. Функциональные требования

### 1.1 Требования к публичной части

1. Система должна предоставлять главную страницу с кратким описанием студии, ключевыми услугами, преимуществами и призывом к отправке заявки.
2. Система должна предоставлять страницу каталога услуг с карточками услуг, базовой стоимостью и сроками выполнения.
3. Система должна предоставлять детальную страницу услуги с полным описанием, этапами работ и кнопкой перехода к заявке.
4. Система должна предоставлять страницу портфолио с кейсами, превью и кратким описанием проектов.
5. Система должна предоставлять поиск по услугам по названию и ключевым словам.
6. Система должна предоставлять фильтрацию каталога по категории и диапазону цены.
7. Гость должен иметь доступ к просмотру публичного контента без регистрации.

### 1.2 Требования к личному кабинету

1. Система должна поддерживать авторизацию пользователя по логину и паролю.
2. Авторизованный пользователь должен иметь доступ к личному кабинету.
3. Пользователь должен видеть список своих заявок с датой, услугой и текущим статусом.
4. Пользователь должен иметь возможность открыть детальную информацию по каждой заявке.
5. Система должна отображать историю изменения статусов заявки (минимум: текущий статус и дата обновления).

### 1.3 Требования к админ-панели

1. Администратор должен иметь доступ к отдельной административной зоне.
2. Администратор должен иметь возможность выполнять CRUD-операции для услуг.
3. Администратор должен иметь возможность выполнять CRUD-операции для портфолио.
4. Администратор должен видеть список входящих заявок с фильтрацией по статусу.
5. Администратор должен иметь возможность изменять статус заявки.
6. Администратор должен иметь возможность просматривать карточку заявки с контактными данными и комментариями.

### 1.4 Требования к AJAX

1. **Фильтрация каталога:** обновление списка услуг без перезагрузки страницы при изменении фильтров.
2. **Отправка формы:** асинхронная отправка заявки с валидацией и отображением результата.
3. **Обновление статуса:** асинхронное изменение статуса заявки в админ-панели с мгновенным обновлением интерфейса.

Примеры целевых AJAX-вызовов (как ориентир для проектирования):

- `fetch('/api/services?category=landing')`
- `fetch('/api/applications', { method: 'POST' })`
- `fetch('/api/applications/1/status', { method: 'PATCH' })`

## 2. Нефункциональные требования

### 2.1 Адаптивность

1. Интерфейс должен корректно отображаться на desktop, tablet и mobile.
2. Ключевые действия (поиск, фильтр, заявка) должны оставаться доступными на мобильных устройствах.

### 2.2 Безопасность

1. Доступ к личному кабинету и админке должен быть ограничен аутентификацией и ролевой моделью.
2. Пользовательский ввод должен проходить серверную валидацию.
3. Для защиты от SQL-инъекций должны использоваться только параметризованные запросы.
4. Пароли должны храниться только в виде хеша (bcrypt).

### 2.3 Работа с БД

1. Подключение к PostgreSQL должно выполняться через `pg` (`node-postgres`).
2. Все операции чтения/записи должны использовать параметризованные SQL-запросы.

### 2.4 Управление авторизацией

1. Авторизация должна поддерживаться через JWT-токены.
2. Должна быть предусмотрена проверка прав доступа к защищенным разделам и admin-эндпоинтам.
3. Выход пользователя должен очищать локальные токены сессии.

### 2.5 Производительность

1. Время загрузки ключевых страниц при стандартном канале связи должно быть минимизировано за счет оптимизации статических ресурсов.
2. Размер изображений для каталога и портфолио должен быть оптимизирован для веба.
3. Асинхронные операции не должны блокировать интерфейс пользователя.
